# Архитектура модуля `local.myvisits`

## Цель решения

Модуль нужен для  фиксации просмотров страниц внутри Bitrix 

модуль встраивается в Bitrix через событие `OnEpilog`,  HTTP-запрос и сохраняет только полезные просмотры страниц.

## Общая схема работы

```
HTTP-запрос
  -> Bitrix обрабатывает страницу
  -> событие main:OnEpilog
  -> \MyVisits\VisitLogger::onEpilog()
  -> фильтрация служебных запросов
  -> VisitTable::add(...)
  -> запись в b_visits_log
  -> компонент myvisits:logviewer читает последние записи и выводит таблицу
```

## Основные части модуля

### 1. Установщик модуля

Файл: `local/modules/local.myvisits/install/index.php`

Отвечает за:

- регистрацию модуля `local.myvisits`
- регистрацию обработчика `main:OnEpilog`
- копирование публичного файла `myvisitors.php`
- удаление `myvisitors.php` при деинсталляции
- снятие регистрации обработчика и самого модуля

Почему это вынесено в установщик:

- дефолт для модулей битрикс
- простая интеграция исключение человеческого фактора
- модуль можно безопасно переустановить без ручных правок сайта

## 2. Точка входа модуля

Файл: `local/modules/local.myvisits/include.php`

Назначение:

- подключает классы модуля при `Loader::includeModule('local.myvisits')`

## 3. Логгер посещений

Файл: `local/modules/local.myvisits/lib/VisitLogger.php`

Назначение:

- получает текущий HTTP-запрос
- определяет URL страницы
- отсеивает системные и шумовые запросы
- сохраняет запись в таблицу через ORM

Почему выбран `OnEpilog`

Были возможны разные стратегии:

1. Ловить запросы вручную в шаблонах сайта
2. Использовать веб-серверные access log
3. Запускать отдельный агент или cron
4. Подписаться на событие Bitrix

Выбран `OnEpilog`, потому что:

- не нужно менять шаблоны и страницы сайта
- модуль работает просто и понятно
- к моменту вызова уже доступен корректный объект запроса
- можно получить URL, IP и referrer в контексте реального запроса
- интеграция остаётся дефолтной для Bitrix

Почему не были выбраны другие подходы:

- ручная вставка в шаблоны нарушает DRY и плохо масштабируется
- access log веб-сервера сложнее связывать с логикой приложения
- cron и агенты хороши для фоновых задач, но не для фиксации живого HTTP-запроса

## 4. Фильтрация запросов

В `VisitLogger` заложена стратегия сокращения шума.

Не логируются:

- служебные пути `/bitrix/`, `/upload/`, `/ajax/`
- `POST`-запросы
- AJAX-запросы
- обращения к статическим файлам

(как и просили в тз)

## 5. Слой хранения данных

Файл: `local/modules/local.myvisits/lib/VisitsLog/VisitTable.php`

Используется D7 ORM:

- таблица: `b_visits_log`
- класс: `MyVisits\VisitsLog\VisitTable`

Поля:

- `ID`
- `TIMESTAMP_X`
- `IP_ADDRESS`
- `PAGE_URL`
- `REFERER`


## 6. Компонент вывода

Файл: `local/components/myvisits/logviewer/component.php`

Назначение:

- подключает модуль `local.myvisits`
- получает последние записи из `VisitTable`
- передаёт данные в шаблон компонента


## 7. Публичная страница

Файл: `myvisitors.php`

Это  обёртка над компонентом `myvisits:logviewer`.

- модуль сразу можно проверить после установки
- не нужно вручную создавать тестовую страницу
- упрощает запуск

 файл удаляется при деинсталляции:

- модуль должен убирать за собой публичные артефакты
- это уменьшает риск “висячих” страниц после удаления решения
